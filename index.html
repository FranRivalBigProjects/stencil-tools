What is the Stencil compiler

The Stencil compiler is a build-time tool — not a runtime library. It takes TypeScript source files annotated with Stencil decorators (@Component, @Prop, @State, @Event, @Watch) and compiles them into standard Web Components (Custom Elements v1 spec) that run natively in all modern browsers without any framework dependency at runtime.

Unlike React or Vue which ship a runtime to the browser, Stencil components compile down to vanilla JavaScript custom elements. The compiler itself (~40MB npm package) never reaches the user's browser — only the compiled output does.

Stencil compiler pipeline

1. TypeScript Transformation

Stencil uses the TypeScript compiler API to strip decorators and replace them with Web Component lifecycle methods. @Prop() becomes observed attributes + getters/setters. @State() triggers re-render on change. @Event() becomes CustomEvent dispatching.

2. JSX Compilation

Stencil's JSX compiles to h() hyperscript function calls (similar to Mithril/Preact). The virtual DOM diffing runs at runtime inside the compiled component — only changed DOM nodes are updated. Shadow DOM encapsulates styles automatically.

3. Dependency Graph Analysis

The compiler builds a dependency graph of all components. my-card that uses my-button forms a tree. This graph drives code splitting — each component and its dependencies bundle independently, enabling per-component lazy loading.

4. Rollup Bundling

Stencil uses Rollup internally for bundling. Each component gets its own chunk. Common utilities are deduplicated into shared chunks. Tree-shaking removes unused code. The output is ES modules — no Webpack, no Babel required.

5. Output Target Generation

A single Stencil project can produce multiple output formats simultaneously based on outputTargets in stencil.config.ts. Each target (lazy, dist, custom-elements) is optimized for a different consumption pattern.

6. Type Declaration Generation

The compiler automatically generates .d.ts type declarations for all components, including prop types, event payloads, and method signatures. Framework wrappers (React, Vue, Angular) use these types for auto-complete and type checking.

Stencil output targets

The outputTargets array in stencil.config.ts controls what the compiler produces. Multiple targets can be defined in one project.

Stencil compiler decorators

@Component({ tag, styleUrl, shadow })
Registers the class as a custom element. tag defines the HTML tag name (must contain a hyphen). shadow: true enables Shadow DOM encapsulation. styleUrl path is resolved relative to the component file and inlined during compilation. scoped: true uses scoped CSS instead of Shadow DOM for style isolation without a true shadow root.

@Prop() — external properties
Declares an external property that maps to an HTML attribute. Props are one-way: parent sets them, child reads them. @Prop({ mutable: true }) allows child mutation. @Prop({ reflect: true }) syncs the property back to the attribute. The compiler generates observedAttributes and attributeChangedCallback for all reflected props automatically.

@State() — internal state
Marks a class property as internal state. Any mutation triggers a re-render. Unlike @Prop(), State is not reflected to attributes and is not accessible from outside the component. Use @State() for toggles, loading states, computed values, and internal data that shouldn't be part of the component's public API.

@Event() and @Listen()
@Event() creates a CustomEvent dispatcher. @Event() myClick: EventEmitter<string>; lets you call this.myClick.emit('value') which fires a bubbling CustomEvent. @Listen('click') attaches event listeners that are automatically removed when the component disconnects. @Listen('resize', { target: 'window' }) listens on the window object.

Frequently Asked Questions
The Stencil compiler is a build-time tool that transforms TypeScript decorated with Stencil decorators (@Component, @Prop, @State, @Event) into standard Web Components (Custom Elements v1). It uses the TypeScript compiler API for transformation and Rollup for bundling. The compiler produces lazy-loaded chunks, type declarations, and multiple output formats. Nothing from the compiler ships to the browser — only the compiled Web Components do.
Stencil is a higher-level, opinionated compiler specifically for Web Components. It uses Rollup (not Webpack) internally and handles TypeScript transformation itself without Babel. Stencil automatically handles code splitting per component, generates multiple output targets simultaneously, produces type declarations, and manages the Web Component registration boilerplate. You don't configure Rollup or TypeScript directly — Stencil's stencil.config.ts provides a higher-level API.
dist-lazy generates a loader script that lazy-loads component chunks on first use — ideal for CDN distribution or adding components via a script tag to any webpage. custom-elements outputs each component as a standalone ES module with no lazy loading — ideal when building framework wrappers (React, Vue, Angular) that import specific components directly. Most component libraries use both: dist-lazy for CDN users and custom-elements for framework integration.
No. Stencil is a self-contained compiler — it uses Rollup internally and you run it with stencil build or stencil build --watch. No separate Webpack, Vite, or Parcel configuration is needed. The Stencil dev server (stencil build --dev --watch --serve) handles HMR during development. However, if you embed Stencil-compiled components into an existing Vite or Webpack application, you just import the compiled output like any ES module.
When shadow: true in @Component(), the Stencil compiler wraps the component's DOM in a Shadow Root, fully encapsulating its CSS from the page. Component styles are inlined into the Shadow DOM at runtime. When scoped: true is used instead, Stencil adds unique class-based attribute selectors to all styles — providing style isolation without a true shadow root, which improves compatibility with server-side rendering.
